서버 시간 기반 타이머를 클라이언트 UI에 표시하고, 실시간 전투 데이터를 복제용 스냅샷으로 분리하여 네트워크 갱신 빈도를 제어한 과정

1. 오늘 구현한 내용

GameState의 상태를 나누고, 각 상태에 맞는 제한 시간을 설정했다. 서버에서는 타이머의 기준이 되는 시작 시간과 종료 시간을 관리하고, 클라이언트에서는 복제받은 시간 정보를 이용해 남은 시간을 계산한다.

이후 플레이어의 데미지, 킬, 아이템 획득, 생존 시간 등을 수집하는 전투정보 구조체를 만들었다. 이 과정에서 연발 사격처럼 값이 매우 자주 변경되는 데이터를 그대로 복제하면 갱신 빈도가 높아질 수 있다는 문제를 발견했다.

2. 서버 시간 기반 타이머

서버가 정의하는 값

서버는 상태가 지속될 시간의 길이를 정한다.

// 예: 게임을 50초 동안 진행
const float Duration = 50.0f;

GameState에는 시간 정보를 담는 구조체를 선언하고 복제한다.

USTRUCT(BlueprintType)
struct FPhaseTimerData
{
    GENERATED_BODY()

    UPROPERTY()
    float StartServerTime = 0.0f;

    UPROPERTY()
    float Duration = 0.0f;

    UPROPERTY()
    float EndServerTime = 0.0f;
};

UPROPERTY(ReplicatedUsing = OnRep_PhaseTimerData)
FPhaseTimerData PhaseTimerData;

클라이언트 UI 갱신

클라이언트가 시간정보 구조체를 복제받는다.
OnRep_PhaseTimerData()에서 델리게이트를 Broadcast한다.
위젯은 델리게이트에 시간 갱신 함수를 바인딩한다.
위젯 내부에서 0.1초 간격의 로컬 타이머를 실행한다.
매 호출마다 서버 기준 현재 시간을 구해 남은 시간을 표시한다.
float ATestGameState::GetRemainTime() const
{
    return FMath::Max(
        0.0f,
        PhaseTimerData.EndServerTime - GetServerWorldTimeSeconds()
    );
}
핵심: 남은 시간을 매 프레임 서버에서 전송하는 것이 아니다. 서버는 시작 시간·지속 시간·종료 시간만 복제하고, 클라이언트는 자신의 로컬 타이머로 화면을 갱신한다.

패킷 지연과 서버 시간

처음에는 시간정보 패킷이 늦게 도착하면 서버와 클라이언트의 시간이 크게 어긋날 것으로 생각했다. 하지만 GetServerWorldTimeSeconds()는 클라이언트에서 서버 시간과 로컬 시간의 차이를 보정한 추정값을 반환한다.

따라서 패킷이 늦게 도착하더라도 종료 시각에서 현재 서버 추정 시간을 빼면, 패킷이 이동하는 동안 이미 흐른 시간이 계산에 반영된다.

네트워크 지연, 프레임 지연, 패킷 손실 때문에 서버와 클라이언트의 시간이 완전히 일치하지는 않는다. 실제 상태 전환과 승패 판정은 반드시 서버가 담당하고, 클라이언트 계산값은 UI 표시용으로만 사용한다.

3. GameState 상태 분기

게임 진행 상태를 Enum으로 나누었다.

UENUM(BlueprintType)
enum class EGameState : uint8
{
    None,
    Lobby,
    Playing
};

플레이어가 접속하면 StartLobby()를 호출한다. 이 함수에서 로비 상태와 시간정보 구조체를 설정하고, 서버의 상태 확인 타이머를 시작한다. 

void ATestGameState::StartLobby()
{
    if (!HasAuthority()) return;

    const float Now = GetServerWorldTimeSeconds();
    const float LobbyDuration = 50.0f;

    PhaseTimerData.GameStateEnum = EGameState::Lobby;
    PhaseTimerData.StartServerTime = Now;
    PhaseTimerData.Duration = LobbyDuration;
    PhaseTimerData.EndServerTime = Now + LobbyDuration;

    GetWorldTimerManager().SetTimer(
        PhaseCheckTimerHandle,
        this,
        &ThisClass::CheckPhaseTime,
        0.5f,
        true
    );
}
void ATestGameState::CheckPhaseTime()
{
    if (!HasAuthority()) return;
    if (GetRemainTime() > 0.0f) return;

    switch (PhaseTimerData.GameStateEnum)
    {
    case EGameState::Lobby:
        MoveToStage();
        break;

    case EGameState::Playing:
        MoveToLobby();
        break;

    default:
        break;
    }
}
상태 남은 시간이 0일 때
Lobby MoveToStage() 실행
Playing MoveToLobby() 실행

4. 플레이어 전투정보 수집

Stage로 이동한 뒤에는 플레이어별 전투정보를 수집한다. 구조체에는 가한 피해, 받은 피해, 킬 수, 획득 아이템 수, 생존 시간, 점수, 순위 등을 저장한다. 

USTRUCT(BlueprintType)
struct FCollectCombatInfo
{
    GENERATED_BODY()

    UPROPERTY()
    float DamageDealt = 0.0f;

    UPROPERTY()
    float DamageTaken = 0.0f;

    UPROPERTY()
    int32 KillCount = 0;

    UPROPERTY()
    int32 LootCount = 0;

    UPROPERTY()
    float SurviveTime = 0.0f;

    UPROPERTY()
    int32 Score = 0;

    UPROPERTY()
    int32 Rank = 0;
};

TakenDamage()처럼 서버에서 실행되는 함수에서 피해량을 기록한다. 공격자의 PlayerState에는 가한 피해를, 피격자의 PlayerState에는 받은 피해를 누적한다.

5. Replicated 구조체를 직접 변경했을 때의 문제

처음에는 전투정보 구조체 자체를 ReplicatedUsing으로 선언하고, 데미지가 발생할 때마다 직접 값을 변경하려 했다. 

UPROPERTY(ReplicatedUsing = OnRep_PlayerCombatInfo)
FCollectCombatInfo PlayerCombatInfo;

PlayerCombatInfo.DamageTaken += ActualDamage;

클라이언트에서는 OnRep_PlayerCombatInfo()에서 Broadcast하고, UI가 이 델리게이트를 구독해 화면을 갱신하도록 만들 수 있다. 

void APlayerCharacterState::OnRep_PlayerCombatInfo(
    const FCollectCombatInfo& OldCombatInfo)
{
    OnPlayerCombatInfoChanged.Broadcast();
}
문제는 연발 사격과 다수의 플레이어다. 복제 대상 변수 자체를 매 타격마다 변경하면, 네트워크 복제 주기에 따라 변경된 구조체가 자주 전송될 수 있다. 단순히 UI Broadcast만 0.1초마다 제한해도 복제 변수의 변경은 이미 발생한 상태다.

Dirty Flag는 변경 여부를 확인하거나 불필요한 계산을 막는 데는 유용하지만, 이미 변경된 Replicated 변수의 네트워크 감지를 취소하지는 못한다.

6. 일반 데이터와 복제용 스냅샷 분리

복제 빈도를 제어하려면 실시간으로 누적되는 서버 전용 데이터와, 클라이언트에 전달할 복제용 데이터를 분리해야 한다. 

// 서버에서 실시간으로 변경되는 원본
UPROPERTY()
FCollectCombatInfo RuntimeCombatInfo;

// 클라이언트로 전달되는 복제용 스냅샷
UPROPERTY(ReplicatedUsing = OnRep_PlayerCombatInfo)
FCollectCombatInfo PlayerCombatInfo;

// 마지막 스냅샷 이후 변경 여부
bool bCombatInfoDirty = false;

전투 이벤트에서는 Runtime 데이터만 변경

void APlayerCharacterState::AddDamageTaken(float Damage)
{
    if (!HasAuthority()) return;

    RuntimeCombatInfo.DamageTaken += Damage;
    bCombatInfoDirty = true;
}

데미지를 주거나 받고, 아이템을 획득하거나 생존 시간이 바뀔 때마다 RuntimeCombatInfo만 수정한다. 이 변수는 복제 대상이 아니므로 값이 자주 바뀌어도 직접적인 복제 패킷이 발생하지 않는다.

0.1초 타이머에서 Score와 Rank 계산

GameState의 0.1초 타이머 함수에서 각 PlayerState의 생존 시간과 점수를 계산한다. 점수 계산이 끝난 뒤 정렬하고 Rank를 지정한다. 

void ATestGameState::CalculatePlayerRank()
{
    if (!HasAuthority()) return;

    TArray<APlayerCharacterState*> Players;

    for (APlayerCharacterState* State : LobbyPlayerStates)
    {
        if (!IsValid(State)) continue;
        if (!State->GetIsParticipatedInThisRound()) continue;

        State->UpdateSurviveTime();
        State->CalculateCombatScore();
        Players.Add(State);
    }

    Players.Sort(
        [](const APlayerCharacterState& A,
           const APlayerCharacterState& B)
        {
            return A.GetRuntimeCombatInfo().Score
                 > B.GetRuntimeCombatInfo().Score;
        });

    int32 CurrentRank = 1;

    for (int32 Index = 0; Index < Players.Num(); ++Index)
    {
        APlayerCharacterState* CurrentPS = Players[Index];
        if (!IsValid(CurrentPS)) continue;

        if (Index > 0)
        {
            const APlayerCharacterState* PrevPS = Players[Index - 1];

            if (IsValid(PrevPS) &&
                CurrentPS->GetRuntimeCombatInfo().Score
                < PrevPS->GetRuntimeCombatInfo().Score)
            {
                CurrentRank = Index + 1;
            }
        }

        CurrentPS->SetRuntimeRank(CurrentRank);
    }

    for (APlayerCharacterState* State : Players)
    {
        if (IsValid(State))
        {
            State->CommitCombatInfoSnapshot();
        }
    }
}

계산이 끝난 뒤 복제용 구조체에 한 번 대입

void APlayerCharacterState::CommitCombatInfoSnapshot()
{
    if (!HasAuthority()) return;
    if (!bCombatInfoDirty) return;

    bCombatInfoDirty = false;
    PlayerCombatInfo = RuntimeCombatInfo;

    // Listen Server의 로컬 UI가 필요할 때 사용
    OnPlayerCombatInfoChanged.Broadcast();

    // 일반적으로는 다음 네트워크 갱신을 기다린다.
    // ForceNetUpdate();
}

Dedicated Server에서는 로컬 UI가 없기 때문에 서버 Broadcast가 실제 UI를 갱신하지 않는다. 클라이언트 UI는 복제 데이터가 도착한 뒤 OnRep_PlayerCombatInfo()에서 갱신된다.

이 구조에서는 0.1초 동안 데미지 이벤트가 수십 번 발생해도 복제용 PlayerCombatInfo는 Commit 시점에 한 번만 변경된다. 전투 이벤트 횟수와 네트워크 스냅샷 갱신 횟수가 분리된다.
방식 데미지 발생 시 복제 변수 변경 역할
Replicated 변수 직접 수정 매 타격마다 수정 매 타격마다 가능 구현은 단순하지만 빈번한 변경 발생
Dirty Flag만 사용 매 타격마다 수정 이미 변경됨 계산 및 Broadcast 제어
Runtime + Snapshot Runtime만 수정 0.1초 Commit 시 수정 복제 변수 변경 빈도 제어

7. 최종 동작 흐름

데미지·킬·아이템 획득 이벤트가 서버에서 발생한다.
RuntimeCombatInfo에 실시간 데이터가 누적된다.
bCombatInfoDirty를 true로 변경한다.
GameState의 0.1초 타이머가 생존 시간과 Score를 계산한다.
Runtime Score를 기준으로 정렬하고 Rank를 지정한다.
CommitCombatInfoSnapshot()에서 복제용 구조체에 한 번 복사한다.
다음 네트워크 갱신에서 클라이언트가 스냅샷을 받는다.
OnRep_PlayerCombatInfo()가 Broadcast하고 UI를 갱신한다.
전투 이벤트 여러 번
        ↓
RuntimeCombatInfo 실시간 누적
        ↓
0.1초 타이머
        ↓
SurviveTime / Score / Rank 계산
        ↓
PlayerCombatInfo = RuntimeCombatInfo
        ↓
Replication
        ↓
OnRep
        ↓
클라이언트 UI 갱신

8. 배운 점

  • 서버는 상태 전환과 게임 판정을 담당하고, 클라이언트 타이머는 UI 표시용으로 사용한다.
  • 남은 시간을 계속 복제하기보다 시작·종료 시각을 한 번 전달하고 로컬에서 계산하는 편이 효율적이다.
  • GetServerWorldTimeSeconds()는 클라이언트에서도 서버 기준 시간을 추정할 수 있게 해준다.
  • Broadcast는 로컬 함수 호출이며, 그 자체가 네트워크 패킷을 전송하는 것은 아니다.
  • Dirty Flag만으로는 이미 변경된 Replicated 변수의 복제 감지를 막을 수 없다.
  • 빈번하게 바뀌는 데이터는 서버 전용 Runtime 값과 복제용 Snapshot으로 분리할 수 있다.
  • 점수를 먼저 계산한 뒤 정렬하고, 마지막에 Rank를 지정해야 한다.
  • 0.1초마다 Commit한다고 해서 OnRep 도착 간격이 정확히 0.1초로 보장되는 것은 아니다. 실제 전송은 Actor의 네트워크 갱신 주기와 네트워크 상태의 영향을 받는다.
